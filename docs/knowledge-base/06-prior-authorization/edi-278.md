# EDI 278 — Health Care Services Review

## Overview

The EDI 278 transaction is the HIPAA-compliant electronic data interchange (EDI) standard used for submitting prior authorization requests and receiving responses. It is part of the X12N (Insurance) transaction set family. The 278 transaction consists of a **Request** (278R) sent by the provider to the payer, and a **Response** (278R) returned by the payer with the authorization decision.

The 278 replaces paper/fax/phone authorization submissions and enables automated, system-to-system authorization processing. It is commonly used for:
- Inpatient and outpatient treatment authorization
- Specialty drug pre-authorization
- DME and supply authorization
- Imaging and diagnostic testing authorization
- Physical therapy and rehabilitation authorization
- Concurrent review and continued stay certification

## 1. 278 Request Structure (Request for Review)

The 278 request uses the X12N 278 transaction set. Below is the hierarchical structure with the key segments and loops.

### Transaction Header

```
ISA*00*          *00*          *ZZ*SENDERID       *ZZ*RECEIVERID      *200601*1200*^*00501*000000001*0*T*:~
GS*HI*SENDERID*RECEIVERID*20250610*1200*1*X*005010X217~
ST*278*0001~
BHT*0001*13*AUTHREQ001*20250610*1200~
```

| Segment | Element | Description |
|---|---|---|
| **ISA** | Interchange Control Header | Sender/Receiver IDs, date, time, control number |
| **GS** | Functional Group Header | Functional group identifier (HI = Health Care Services Review), sender/receiver |
| **ST** | Transaction Set Header | 278 = Health Care Services Review |
| **BHT** | Beginning of Hierarchical Transaction | BHT02 = 13 (Request); BHT03 = Authorization Number (optional in request, required in response) |

### BHT Segment Details

- **BHT01**: Hierarchical structure code — "0001" (single-level hierarchy)
- **BHT02**: Purpose code — "13" (Request) or "11" (Response)
- **BHT03**: Reference identification — Unique transaction ID (request) or authorization number (response)
- **BHT04**: Transaction date
- **BHT05**: Transaction time

### Loop 2000A — Information Source (Payer Level)

```
HL*1**20*1~
```

- **HL01**: Hierarchical ID number (1)
- **HL02**: Hierarchical parent ID number (blank if top-level)
- **HL03**: Hierarchical level code — **20** = Information Source (Payer)
- **HL04**: Child indicator — "1" = has subordinate levels

**NM1** segment (Payer Name):

```
NM1*PR*2*PAYER NAME*****PI*PAYERID~
```

- **NM101**: Entity code — "PR" = Payer
- **NM102**: Type — "1" (Person) or "2" (Non-Person Entity)
- **NM103**: Payer name
- **NM108**: ID qualifier — "PI" = Payer ID
- **NM109**: Payer identifier (the payer's unique ID)

### Loop 2000B — Information Receiver (Provider Level)

```
HL*2*1*21*1~
```

- **HL01**: Hierarchical ID number (2)
- **HL02**: Hierarchical parent ID number (1 — parent is the payer)
- **HL03**: Hierarchical level code — **21** = Information Receiver (Provider)
- **HL04**: Child indicator — "1" = has subordinate levels

**NM1** segment (Provider Name):

```
NM1*1P*2*PROVIDER NAME*****XX*1234567890~
```

- **NM101**: Entity code — "1P" = Provider
- **NM102**: Type — "2" (Non-Person Entity for facility/group)
- **NM103**: Provider name
- **NM108**: ID qualifier — "XX" = NPI
- **NM109**: Provider NPI

**PRV** segment (Provider Information):

```
PRV*RF*PXC*207Q00000X~
```

- **PRV01**: Provider code — "RF" = Referring Provider or "AD" = Attending, "PE" = Performing
- **PRV02**: Reference identification qualifier — "PXC" = Taxonomy Code
- **PRV03**: Taxonomy code

### Loop 2000B — UM (Utilization Management)

```
UM*HS*EM*20*1*20250610**1~
```

- **UM01**: Request category code
  - `HS` = Health Services Review (most common)
  - `CA` = Certification of Admission
  - `CC` = Continued Stay Certification
  - `CE` = Certification of Extension
  - `CT` = Certification of Termination
  - `SC` = Specialty Certification
  - `PT` = Partial Certification
- **UM02**: Certificate/payer authorization number (used in response)
- **UM03**: Service type code:
  - `EM` = Emergency
  - `AC` = Ambulatory Care
  - `CR` = Comprehensive Review
  - `MH` = Mental Health
  - `PT` = Physical Therapy
  - `1` = Medical Care, `2` = Surgical, `3` = Consultation, etc.
  - `98` = Professional (Physician), `47` = Hospital
- **UM04**: Certification/Authorization type code:
  - `1` = Initial (first request)
  - `2` = Renewal (renewal of existing auth)
  - `3` = Extension
  - `4` = Change/Modification
- **UM05**: Number of units requested (e.g., number of visits)
- **UM06**: Date of service (from date)
- **UM07**: Date of service (to date)
- **UM08**: Authorization expiration date
- **UM09**: Quantity and type — "1/D" (one per day), "2/W" (two per week), "10/M" (ten per month)

### Loop 2000B — DMG (Demographic Information)

```
DMG*D8*19800115*F~
```

- **DMG01**: Date format — "D8" = Date (CCYYMMDD)
- **DMG02**: Patient date of birth
- **DMG03**: Gender — "M" (Male), "F" (Female), "U" (Unknown)

### Loop 2000B — INS (Subscriber Information)

```
INS*Y*18*001*25*A***AC~
```

- **INS01**: Insured indicator — "Y" (Yes, the subscriber is the patient) or "N" (No, patient is a dependent)
- **INS02**: Relationship code:
  - `18` = Self
  - `19` = Spouse
  - `20` = Child
  - `21` = Other Dependent
  - `39` = Organ Donor
  - `40` = Cadaver Donor
- **INS03**: Employment status code — `001` = Employed Full-Time
- **INS04**: Student status code — `25` = Full-Time Student
- **INS05**: Patient location code
- **INS06**: Date of death
- **INS07**: Policy code — `AC` = Active Coverage

**HI** segment (Health Diagnosis):

```
HI*BK:J02*BF:J030~
```

- **HI01** onward: Diagnosis codes with qualifiers
  - `BK` = Principal diagnosis
  - `BF` = Other diagnosis
  - `AB` = Admitting diagnosis
  - `AP` = Allergic condition
  - `DR` = Diagnostic related group (DRG)
- Format: Qualifier + colon + ICD-10-CM code

**DTP** segment (Dates — Certification Period):

```
DTP*291*D8*20250610~
```

- **DTP01**: Date/Time qualifier
  - `291` = Certification Period Start
  - `292` = Certification Period End
- **DTP02**: Date format — `D8` (CCYYMMDD)
- **DTP03**: Date value

**DTP** segment (Admission/Start Date):

```
DTP*435*D8*20250610~
```

- **DTP01**: `435` = Admission Date / Start of Service Date

### Loop 2000C — Dependent Level (if patient is a dependent)

```
HL*3*2*22*0~
```

- **HL01**: Hierarchical ID number (3)
- **HL02**: Hierarchical parent ID number (2 — parent is the provider)
- **HL03**: Hierarchical level code — **22** = Subscriber/Dependent
- **HL04**: Child indicator — "0" = no further levels

**Patient Name (NM1)**:

```
NM1*QC*1*PATIENT*FIRST****MI*123456789~
```

- **NM101**: `QC` = Patient
- **NM108**: `MI` = Member ID or `II` = Member identification number
- **NM109**: Patient identification number (subscriber-dependent ID)

**REF** segment (Patient Account Number):

```
REF*EJ*PATIENTACCT001~
```

- **REF01**: `EJ` = Patient Account Number
- **REF02**: Patient account number (provider's internal account number)

### Loop 2000D — Service Level (SV segments)

For each service/procedure being requested:

```
SV*HC:99213*45*UN*1*1**3:1~
```

- **SV101**: Product/Service ID — Qualifier + colon + code
  - `HC` = HCPCS/CPT
  - `N4` = National Drug Code (NDC)
  - `CJ` = Revenue code (for institutional)
- **SV102**: Line item charge amount (optional)
- **SV103**: Unit basis — `UN` = Units
- **SV104**: Quantity — number of units requested
- **SV105**: Place of service code
  - `11` = Office
  - `21` = Inpatient Hospital
  - `22` = Outpatient Hospital
  - `23` = Emergency Room
  - `24` = Ambulatory Surgical Center
- **SV106**: Level of care code (optional)
- **SV107**: Diagnosis pointer — references the HI segment diagnosis positions

**DTP** segments for service-level dates:

```
DTP*472*D8*20250615~
```

- `472` = Service Date

## 3. 278 Response Structure (Response to Review)

The 278 response follows the same hierarchical structure but uses BHT02 = "11" (Response) and includes the authorization decision in the UM segment.

### Key Differences in Response

| Aspect | Request | Response |
|---|---|---|
| **BHT02** | 13 (Request) | 11 (Response) |
| **BHT03** | Transaction reference ID | Authorization number (if approved) |
| **UM02** | Empty | Authorization number assigned by payer |
| **UM03** | Service type code | (Same as request) |
| **UM01** | Request category | (Same as request) |
| **UM05-UM08** | Requested values | Approved values (may differ from request) |

### UM Segment in Response — Status Codes

```
UM*HS*AUTH12345*EM*1*8*20250610*20251231~
```

The **UM01** element in response may use different service type codes. More importantly:

| UM Element | Description | Response Values |
|---|---|---|
| **UM01** | Request category | Same as request |
| **UM02** | Authorization number | Payer-assigned authorization reference number |
| **UM03** | Service type | Same as request |
| **UM04** | **Certification type code / Status** | See status table below |
| **UM05** | Units approved | Number of visits/units approved |
| **UM06** | Certification period from | Start date of authorized period |
| **UM07** | Certification period to | End date of authorized period |

### Certification Type / Status Codes (UM04)

| Code | Status | Description |
|---|---|---|
| **1** | Accepted / Approved | Authorization granted as requested |
| **2** | Pended | Additional information required; more time needed |
| **3** | Denied | Authorization denied — service not medically necessary or not covered |
| **4** | Accepted but Modified | Approved with changes (different codes, fewer units, different location) |
| **5** | Accepted — Partial | Some services approved, others denied |
| **6** | Expired | Previous authorization has expired |
| **7** | Approved — No Changes | Same as code 1, explicitly unchanged |
| **8** | Approved — Modified | Same as code 4, approved with changes |
| **A** | Approved as Submitted | Approved exactly as in the request |
| **B** | Approved with Conditions | Approved contingent on additional criteria being met |

### Response Example — Approved

```
ST*278*0002~
BHT*0001*11*AUTH12345*20250610*1201~
HL*1**20*1~
NM1*PR*2*UNITEDHEALTHCARE*****PI*87726~
HL*2*1*21*1~
NM1*1P*2*ACME MEDICAL GROUP*****XX*1234567890~
UM*HS*AUTH12345*EM*1*8*20250610*20251231~
DMG*D8*19750315*M~
INS*Y*18*001~
HI*BK:M545~
DTP*291*D8*20250610~
DTP*292*D8*20251231~
HL*3*2*22*0~
NM1*QC*1*DOE*JOHN****MI*MEM001234~
REF*EJ*ACCT56789~
SV*HC:73721*1200*UN*1*22~
DTP*472*D8*20250615~
SE*16*0002~
```

In this response:
- **BHT03**: Contains the authorization number (AUTH12345)
- **UM02**: Same authorization number (AUTH12345)
- **UM04**: Status "1" = Approved
- **UM05**: 8 units approved (could have been requested at 10 units — payer modified)
- **UM06/UM07**: Authorization valid dates

### Response Example — Denied

```
UM*HS**EM*3**0***~
```

- **UM02**: No auth number (denied)
- **UM04**: Code "3" = Denied
- **UM05**: 0 units approved
- No valid dates

Additional reason codes may appear in **REF** segments following the UM:

```
REF*9N*DENIAL123~
REF*BB*W9001~
```

- **REF01** = `9N`: Denial tracking number
- **REF01** = `BB`: Authorization or certification number being modified
- **REF01** = `EW`: Denial reason code (internal payer code)

## 4. 278 Real-Time vs. Batch Processing

### Real-Time 278

| Aspect | Description |
|---|---|
| **Processing** | Request and response within seconds |
| **Protocol** | Interactive TCP/IP connection or web services API |
| **Use case** | Same-day service scheduling, urgent auth |
| **Clearinghouse** | Many clearinghouses support real-time 278 (Change Healthcare, Availity, Navinet) |
| **Volume** | Suitable for individual requests |
| **Cost** | Per-transaction fee typically higher |
| **Advantage** | Immediate decision — can schedule service immediately |
| **Disadvantage** | Less detail in response; may not handle complex cases |

### Batch 278

| Aspect | Description |
|---|---|
| **Processing** | Submitted in batch files; response in hours to days |
| **Protocol** | SFTP, OFTP, or clearinghouse batch portal |
| **Use case** | High-volume, routine authorizations |
| **Clearinghouse** | Standard batch processing via all major clearinghouses |
| **Volume** | Designed for bulk submissions (hundreds or thousands) |
| **Cost** | Lower per-transaction cost |
| **Advantage** | Efficient for scheduled procedures; more detailed response |
| **Disadvantage** | Delayed response; cannot schedule same-day services |

## 9. 278 Error Handling and Troubleshooting

### Common 278 Processing Errors

| Error | Cause | Resolution |
|---|---|---|
| **Transaction rejected (999)** | Bad ISA/GS formatting, invalid trading partner | Check clearinghouse configuration, verify trading partner agreement |
| **UM segment missing** | Required authorization info omitted | Verify UM segment is populated |
| **Invalid taxonomy** | PRV segment taxonomy doesn't match provider | Verify provider taxonomy in the request matches provider file |
| **Diagnosis code error** | Invalid ICD-10 code or wrong qualifier | Use only valid ICD-10-CM codes; use correct HI qualifier (BK, BF, AB) |
| **Member not found** | Member ID doesn't match plan | Verify member ID through eligibility check first |
| **Service not covered** | CPT code not a covered benefit | Check payer medical policy for the code |
| **Duplicate request** | Same auth already on file | Look up existing authorization before submitting new request |
| **Response time out** | Real-time connection timeout | Retry; consider batch for complex requests |
| **Missing SV segment** | Service line info not provided | Ensure at least one SV segment for requested service |
| **Date discrepancy** | Dates in DTP don't match UM dates | Ensure consistency across all date segments |

## 10. Best Practices for 278 Implementation

1. **Verify eligibility before auth**: Always perform a 270/271 eligibility check before submitting a 278 to ensure the member is currently active
2. **Use standard code sets**: Use only valid, current ICD-10-CM, CPT, and HCPCS codes
3. **Include all required segments**: Missing segments cause rejections — validate before submitting
4. **Maintain auth reference tables**: Keep payer-specific 278 requirements mapped (different payers require different data elements)
5. **Handle pend responses**: Set up automated workflows to request additional information when status code 2 is received
6. **Track auth expiry**: Record authorization end dates and set alerts for upcoming expiration
7. **Log all 278 transactions**: Maintain a transaction log for audit and appeal purposes
8. **Batch similar requests**: Group requests by payer for batch submission to reduce per-transaction costs
9. **Use clearinghouse validation tools**: Many clearinghouses offer pre-submission validation for 278 files
10. **Implement real-time for urgent, batch for routine**: Route auth requests appropriately based on urgency

## 11. Q&A Pairs for Agent Training

**Q:** What is the difference between EDI 278 and EDI 270/271?
**A:** EDI 270/271 is used for eligibility and benefit verification — checking if a patient has active coverage, what their benefits include, deductibles, copays, etc. EDI 278 is used for prior authorization — requesting approval for a specific service. The 278 is submitted after eligibility is confirmed and before the service is rendered. Both are HIPAA-mandated standard transactions.

**Q:** What does the UM04 status code "2" mean in a 278 response?
**A:** Status code "2" in the UM04 element means the authorization request is "Pended." This indicates the payer needs additional information before making a decision. Common reasons include insufficient clinical documentation, missing lab results, or the need for medical director review. The provider should respond promptly with the requested information to prevent the auth from stalling or eventually being denied.

**Q:** How does the EDI 278 handle multiple procedures in a single request?
**A:** Multiple procedures are submitted using multiple SV segments within the same transaction. Each SV segment represents one procedure code with its own quantity, place of service, and service date DTP segment. The diagnosis pointer in the SV07 element references the HI segment to link each procedure to the appropriate diagnosis. The payer may approve all, some, or none of the requested services, returning separate SV status information in the response.

**Q:** What is the hierarchical structure of a 278 request?
**A:** The 278 uses a hierarchical structure with four levels: Loop 2000A (Information Source - Payer) is the top level, Loop 2000B (Information Receiver - Provider) is the second level, Loop 2000C (Subscriber/Dependent) is the third level, and Loop 2000D (Service Line) is the fourth level. Each level uses an HL (Hierarchical Level) segment with parent-child references.

**Q:** Can a 278 be used for retrospective authorization?
**A:** Yes, a 278 can be used for retrospective authorization (post-service). The same transaction structure applies, but the provider should ensure the service dates in the DTP segments are in the past. Some payers may accept or reject retro auths through EDI 278. If accepted, the response will include the auth number and approval. If retrospective auth is not accepted by the payer, the transaction may be rejected with a reason code indicating the request is not timely.

**Q:** What is the purpose of the PRV segment in the 278?
**A:** The PRV (Provider Information) segment specifies the provider's taxonomy code, which identifies the provider's specialty (e.g., 207Q00000X for Family Medicine, 2085R0202X for Diagnostic Radiology). This is important because some auth requirements are specialty-specific. The taxonomy code in the 278 must match the provider's NPI taxonomy; otherwise, the payer may reject the request.

**Q:** How is the diagnosis information transmitted in a 278?
**A:** Diagnosis information is transmitted in the HI (Health Diagnosis) segment using qualifiers to identify the diagnosis type. Common qualifiers include BK (Principal Diagnosis), BF (Other Diagnosis), AB (Admitting Diagnosis), and AP (Allergic Condition). Each diagnosis is formatted as qualifier:ICD-10-CM_code separated by the element delimiter. For example: HI*BK:J029*BF:J2090~ indicates principal diagnosis of acute pharyngitis and other diagnosis of unspecified sinusitis.

**Q:** What happens when a 278 transaction is rejected at the interchange level?
**A:** A TA1 (Interchange Acknowledgement) is returned indicating a structural error in the ISA/IEA segments. Common causes include mismatched control numbers, invalid dates, or sender/receiver ID mismatches. Unlike a business-level rejection (claim denied for clinical reason), a TA1 rejection means the payer's system could not process the file at all. The provider must fix the interchange-level error and resubmit the entire file.

**Q:** What is the difference between real-time and batch 278 processing?
**A:** Real-time 278 processing returns a response within seconds and is suitable for urgent or same-day service authorizations. It uses interactive TCP/IP or API connections. Batch processing submits multiple 278 requests in a single file (e.g., 50 requests at once) with responses returned in hours to days. Batch is more cost-effective for routine, scheduled procedures. Real-time has higher per-transaction costs but provides immediate decisions.

**Q:** How does a provider include a specific CPT modifier in a 278 request?
**A:** CPT modifiers are included as part of the SV product/service ID. The format is HC:CPTCODE followed by a colon and modifier. For example, SV*HC:73721*1200*UN*1*22~ for the base code, or to include a modifier, the structure would be SV*HC:73721:26*1200*UN*1*22~. The modifier is appended after the CPT code using the colon delimiter within the HC composite.

**Q:** Can a 278 response include a partial approval?
**A:** Yes. Status code 4 (Accepted but Modified) or 5 (Accepted — Partial) indicates the payer approved some but not all of the requested services. For example, if a provider requested 20 PT visits, the payer might approve only 12. The UM05 element would show 12 as the approved quantity while the request had 20. The SV segment may show which specific service lines were altered. The provider should reconcile the approved services against the requested services and adjust scheduling accordingly.

**Q:** What is the BHT segment used for in a 278 transaction?
**A:** The BHT (Beginning of Hierarchical Transaction) segment identifies the transaction as either a request (BHT02 = 13) or a response (BHT02 = 11). In the request, BHT03 contains a unique transaction reference ID assigned by the sender. In the response, BHT03 contains the authorization number assigned by the payer. BHT04 and BHT05 carry the transaction date and time.

**Q:** How does a 278 handle dependent patients (child covered under parent's plan)?
**A:** For dependent patients, the HL structure extends to Loop 2000C (Dependent Level). The INS segment at Loop 2000B indicates INS02 relationship code 20 (Child), 19 (Spouse), or 21 (Other Dependent). The dependent's name appears at Loop 2010C with NM101 = "QC" (Patient) and a unique subscriber-dependent ID number. This structure allows the payer to identify which specific family member needs authorization.

**Q:** What clearinghouse services support EDI 278?
**A:** Major clearinghouses supporting 278 include Change Healthcare (Emdeon), Availity, Navinet, Office Ally, ZirMed, Waystar, and Trizetto Provider Solutions. Most offer both real-time and batch 278 processing. Many also offer a web portal fallback when EDI submission fails. The provider's practice management system typically connects to a clearinghouse for 278 submissions.

**Q:** What is the relationship between a 278 auth and an 837 claim?
**A:** The authorization number obtained from a successful 278 response must be included in the subsequent 837 claim submission. In the 837P (Professional), the auth number goes in Loop 2300 with REF01 = "G1" (Prior Authorization Number). In the 837I (Institutional), it goes in the same loop structure. If the 837 does not include the valid auth number, the claim will likely deny even though the service was authorized. The two transactions are linked by the authorization number.

**Q:** How long does a standard 278 response remain valid?
**A:** The validity period is specified in the UM06 and UM07 elements (Certification Period Start and End dates) in the 278 response. Typical periods are 30 days for surgery, 60 days for imaging, 1 year for therapy visit blocks, or admission-based for inpatient. The provider must render services within this window. Once the certification period expires, a new 278 request (request code = 2 for Renewal or 3 for Extension) must be submitted.

**Q:** What is the 999 Implementation Acknowledgment and how does it relate to 278?
**A:** The 999 is an EDI acknowledgment transaction that confirms the syntactical validity of a submitted 278 (or any X12 transaction). AK501 = 1 means the transaction was accepted; AK501 = 0 means it was rejected. If rejected, the AK9 segment provides specific error codes identifying which segments are invalid. The 999 is not a business decision — it only confirms the file was readable. The actual auth decision comes in the 278 response.

**Q:** How should a provider handle a status code 3 (Denied) in a 278 response?
**A:** Status code 3 means the authorization was denied. The response may include a REF segment with a denial tracking number and reason code. The provider should: (1) Review the denial reason from additional REF or MSG segments. (2) Determine if it is a clinical denial (not medically necessary), administrative (missing information), or policy-based (not a covered benefit). (3) For clinical denials, prepare a letter of medical necessity and request a peer-to-peer review. (4) For administrative denials, correct the missing information and resubmit. (5) Follow the payer's formal appeal process.

**Q:** What is the SV segment and what information does it contain?
**A:** The SV (Service Line) segment contains the specific service or procedure being requested. It includes: SV101 (Procedure Code with qualifier, e.g., HC:99213); SV102 (Line item charge amount); SV103 (Unit basis — always "UN" for units); SV104 (Quantity — number of units/visits); SV105 (Place of Service code); SV106 (Level of Care code, optional); and SV107 (Diagnosis Pointer — references the HI segment). Multiple SV segments can appear for multi-procedure requests.

**Q:** What is a "crossover" authorization in the context of 278?
**A:** A crossover authorization occurs when a service is approved through one authorization (e.g., inpatient admission) and subsequent related services (e.g., inpatient surgery) are considered covered under that same authorization without needing a separate 278. This is more of a billing concept than a specific 278 feature. The UM01 request category code (HS, CA, CC, CE) defines what type of request is being made, with "CA" = Certification of Admission covering all services within the admission episode.

**Q:** How does the ISA segment differ from the GS segment in a 278 transmission?
**A:** The ISA (Interchange Control Header) wraps the entire EDI interchange containing multiple functional groups. It identifies the sender and receiver at the trading partner level, includes date/time of transmission, and the interchange control number. The GS (Functional Group Header) identifies a specific functional group within the interchange — for 278, the GS01 is always "HI" (Health Care Services Review). One ISA can contain multiple GS groups (e.g., one for 278, one for 837), and one GS can contain multiple ST (transaction) segments.

**Q:** What is the significance of the "D8" date qualifier?
**A:** "D8" in DTP02 and other date-format fields indicates the date is in CCYYMMDD format (e.g., 20250610 for June 10, 2025). Other possible date qualifiers include "RD8" (date range CCYYMMDD-CCYYMMDD), "TM" (time), and "DT" (date and time). The D8 format is the standard for prior authorization dates. Most payers require D8 format; using an unsupported format may cause a reject.
