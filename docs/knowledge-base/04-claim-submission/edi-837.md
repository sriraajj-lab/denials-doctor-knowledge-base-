# EDI 837 (Health Care Claim Transaction)

## Overview

The HIPAA 837 transaction is the electronic format used to submit healthcare claims to payers. It is part of the X12N family of electronic data interchange (EDI) transactions and follows the ASC X12 standards. The 837 transaction replaced the paper CMS-1500 (professional) and UB-04 (institutional) claim forms, enabling automated, standardized claim submission across all payers.

The 837 transaction has three versions:
- **837P (Professional)**: Used by physicians, suppliers, and other professional providers for services billed on the CMS-1500 form.
- **837I (Institutional)**: Used by hospitals, skilled nursing facilities, and other institutional providers for services billed on the UB-04 form.
- **837D (Dental)**: Used by dentists for services billed on the ADA dental claim form.

---

## 1. EDI Envelope Structure

The 837 transaction is transmitted within an EDI envelope that provides routing and control information. The envelope has three levels:

### Interchange Level (ISA-IEA)

The interchange envelope contains the entire transmission — one or more functional groups.

| Segment | Description |
|---------|-------------|
| **ISA** | Interchange Control Header — identifies sender, receiver, control number, date/time, and version |
| **(Functional Groups)** | One or more GS-GE groups |
| **IEA** | Interchange Control Trailer — must match ISA control number |

ISA Segment Fields:

| Field | Position | Description |
|-------|----------|-------------|
| ISA01 | 01 | Authorization Information Qualifier (00 = No auth, 03 = Password) |
| ISA02 | 02 | Authorization Information |
| ISA03 | 03 | Security Information Qualifier (00 = No security, 01 = Password) |
| ISA04 | 04 | Security Information |
| ISA05 | 05 | Interchange Sender ID Qualifier (ZZ = Mutually Defined, 30 = IRS EIN) |
| ISA06 | 06 | Interchange Sender ID |
| ISA07 | 07 | Interchange Receiver ID Qualifier |
| ISA08 | 08 | Interchange Receiver ID |
| ISA09 | 09 | Interchange Date (YYMMDD) |
| ISA10 | 10 | Interchange Time (HHMM) |
| ISA11 | 11 | Interchange Control Standards Identifier (U = USA) |
| ISA12 | 12 | Interchange Control Version Number (00501 = v5010) |
| ISA13 | 13 | Interchange Control Number (must match IEA02) |
| ISA14 | 14 | Acknowledgment Requested (0 = No, 1 = Yes) |
| ISA15 | 15 | Usage Indicator (P = Production, T = Testing) |
| ISA16 | 16 | Component Element Separator |

### Functional Group Level (GS-GE)

The functional group level groups related transaction sets.

| Segment | Description |
|---------|-------------|
| **GS** | Functional Group Header — identifies application sender/receiver, date, time, group control number |
| **(Transaction Sets)** | One or more ST-SE transaction sets |
| **GE** | Functional Group Trailer — number of transaction sets, group control number |

GS Segment Fields:

| Field | Position | Description |
|-------|----------|-------------|
| GS01 | 01 | Functional Identifier Code (HC = Healthcare Claim) |
| GS02 | 02 | Application Sender's Code |
| GS03 | 03 | Application Receiver's Code |
| GS04 | 04 | Date (CCYYMMDD) |
| GS05 | 05 | Time (HHMM) |
| GS06 | 06 | Group Control Number |
| GS07 | 07 | Responsible Agency Code (X = X12) |
| GS08 | 08 | Version/Release/Industry ID Code (005010X222) |

### Transaction Set Level (ST-SE)

The transaction set level contains the actual claim data.

| Segment | Description |
|---------|-------------|
| **ST** | Transaction Set Header — identifies transaction set ID (837) and control number |
| **(Claim Data)** | All segments making up the claim |
| **SE** | Transaction Set Trailer — segment count, control number |

## 3. 837I (Institutional) Key Differences

The 837I is used for institutional claims (hospitals, SNFs, home health) and differs from the 837P in several ways:

| Aspect | 837P (Professional) | 837I (Institutional) |
|--------|---------------------|----------------------|
| **Claim Form** | CMS-1500 | UB-04 |
| **Standard** | 005010X222A1 | 005010X223A2 |
| **Service Line** | SV1 segment | SV2 segment |
| **Revenue Codes** | Optional | Required |
| **Patient Status** | Not used | CL1 segment (patient discharge status) |
| **Admission/Discharge** | Optional DTP | Required DTP |
| **Principal Procedure** | HCP segment | HCP segment required for inpatients |
| **Occurrence Codes** | Not used | CR6, CR5 segments |
| **Value Codes** | Not used | HI segment with value codes |
| **Condition Codes** | Not used | HI segment with condition codes |
| **Bill Type** | Not used | CLM05-1 |
| **DRG** | Not used | HI segment with DRG |
| **Frequency Code** | Not used | CLM05-2 |

### 837I Unique Segments

| Segment | Purpose |
|---------|---------|
| **CL1** | Patient Discharge Status (01 = Discharged to home, 02 = Discharged/transferred to other facility) |
| **HI*ABF** | Principal Diagnosis |
| **HI*ABJ** | Admitting Diagnosis |
| **HI*BBR** | DRG (Diagnosis Related Group) |
| **HI*BEH** | Condition Code |
| **HI*BF** | Occurrence Code |
| **HI*BG** | Occurrence Span |
| **HI*BE** | Value Code and Amount |
| **SV2** | Institutional Service Line (includes Revenue Code, Procedure, Charges) |
| **NTE** | Remark (special handling notes specific to institutional) |

## 4. 837 Submission Modes

### Mode 1: Direct Submission (CMS Direct Data Entry)

- Provider submits claims directly through the CMS Medicare Portal.
- Used by small providers without EDI or clearinghouse capabilities.
- Requires manual entry of each claim.
- No batch submission capability.
- No 999/277CA automated acknowledgments.

### Mode 2: Via Clearinghouse

- Provider submits 837 file to a clearinghouse.
- Clearinghouse translates and routes to payers.
- Clearinghouse returns 999/277CA acknowledgments.
- Supports batch and sometimes real-time submission.
- Most common method for professional providers.

### Mode 3: Via EHR API

- Provider's EHR transmits 837 directly to payer via API.
- Used by integrated health systems and large provider groups.
- Real-time or near-real-time adjudication possible.
- Requires direct payer connectivity and credentialing.

## 5. Batch Submission vs Real-Time Submission

### Batch Submission

| Aspect | Description |
|--------|-------------|
| **Transmission** | Multiple claims grouped into a single 837 file |
| **Acknowledgments** | Asynchronous — 997 immediately, 277CA later (hours to days) |
| **Trading Partner Agreement** | Required — establishes NSF/TRN identifiers and communication protocols |
| **File Format** | Standard ASCII text file or EDI format |
| **Transmission Protocol** | FTP, SFTP, AS2, HTTPS |
| **Acknowledgment** | TA1 (Interchange Ack), 997 (Functional Ack), 277CA (Claim Status) |
| **Suitability** | High-volume submitters, institutional claims, recurring submissions |

### Real-Time Submission

| Aspect | Description |
|--------|-------------|
| **Transmission** | Single claim submitted individually |
| **Acknowledgments** | Synchronous — response in seconds |
| **Trading Partner Agreement** | Varies by payer |
| **File Format** | X12 837 or JSON/XML |
| **Transmission Protocol** | Web services, SOAP, HTTPS |
| **Acknowledgment** | Immediate 999 or payer-specific status |
| **Suitability** | Low-volume submitters, urgent claims, POS verification |

### Transmission Protocols

| Protocol | Description | Security |
|----------|-------------|----------|
| **FTP** | File Transfer Protocol — basic file transfer | Unencrypted (plaintext) |
| **SFTP** | SSH File Transfer Protocol — encrypted file transfer | Encrypted (SSH) |
| **FTPS** | FTP over SSL/TLS | Encrypted (SSL/TLS) |
| **AS2** | Applicability Statement 2 — secure B2B EDI | Encrypted (S/MIME) |
| **HTTPS** | HTTP over TLS | Encrypted (TLS) |
| **SOAP** | Simple Object Access Protocol — web services | Over HTTPS |

## 6. Claim Processing Status

After submission, a claim passes through several status stages:

### Stage 1: Transmission Acceptance

| Status | Description |
|--------|-------------|
| **Accepted** | EDI envelope is valid and accepted by receiver |
| **Rejected** | EDI envelope is invalid (bad ISA, control number duplicate) |

### Stage 2: Functional Acknowledgment (997)

| Status | Description |
|--------|-------------|
| **Accepted** | Transaction set is valid X12 syntax |
| **Rejected** | Transaction set has X12 syntax errors (missing segments, invalid qualifiers) |

### Stage 3: Claim Status (277CA)

| Status | Description |
|--------|-------------|
| **Accepted** | Claim has passed all front-end edits and is in processing |
| **Accepted with Errors** | Claim accepted but certain fields were corrected or are pending |
| **Rejected** | Claim had fatal errors and did not enter adjudication system |
| **Pending** | Claim is held for manual review or additional information |
| **Denied** | Claim has been adjudicated and denied |
| **Partial Payment** | Claim has been adjudicated with partial payment |
| **Paid** | Claim has been adjudicated and paid in full |

## 7. 997 Functional Acknowledgment

The 997 Functional Acknowledgment is the first response to a submitted 837 transaction. It indicates whether the 837 file was syntactically valid.

### 997 Status Codes

| Code | Description |
|------|-------------|
| **A** | Accepted |
| **E** | Accepted but errors noted |
| **M** | Rejected — malformed |
| **R** | Rejected |
| **W** | Accepted but warnings |

### What a 997 Acknowledges

- ISA/IEA envelope structure: valid or invalid.
- GS/GE functional group: valid or invalid.
- ST/SE transaction set: valid or invalid.
- Syntax errors: missing required segments, invalid data types, invalid qualifiers.
- Control number errors: duplicated or out-of-sequence control numbers.

### Important: 997 Is NOT Claim Approval

The 997 acknowledgment means the EDI transmission was technically valid. It does not mean the claims are correct, covered, or will be paid. A claim can pass the 997 but be denied later for medical necessity, eligibility, or benefit limitations. The 997 is a technical acknowledgment only.

## 8. 277CA Claim Status

The 277CA Claim Status response provides detailed claim-level processing status.

### 277CA Status Categories

| Category | Code | Description |
|----------|------|-------------|
| **Claim Accepted** | 1 | Claim accepted for processing |
| **Claim Rejected** | 2 | Claim rejected (format errors at payer level) |
| **Additional Info Required** | 3 | Payer needs more information |
| **Claim Pending** | 4 | Claim in processing queue |
| **Claim Paid** | P | Claim has been paid |
| **Claim Denied** | D | Claim has been denied |
| **Partial Payment** | PA | Claim partially paid |

### 277CA Response Timeline

| Response | Typical Timing | Content |
|----------|---------------|---------|
| **Immediate 997** | Seconds to minutes | Technical acceptance |
| **Initial 277CA** | 24-48 hours | Claim accepted/rejected for processing |
| **Final 277CA** | 5-30 days | Claim processed with payment/denial status |

## 9. Companion Guides

### What Is a Companion Guide?

A companion guide is a document published by each payer that explains how the 837 standard should be implemented for that specific payer. Companion guides do not replace the X12 standard but supplement it with payer-specific requirements.

### What Companion Guides Specify

| Aspect | Examples |
|--------|----------|
| **Required segments** | "We require REF*G1 for all imaging claims" |
| **Optional segments** | "We do not accept the CRC segment" |
| **Situational rules** | "REF*9F is required for workers' comp claims" |
| **Code values** | "Use qualifier ZZ not 08 for provider IDs" |
| **Loop structure** | "Loop 2317B Referring Provider is required for all laboratory claims" |
| **Field lengths** | "Only first 30 characters of patient name will be processed" |
| **Modifier requirements** | "Modifier 25 must be on the E/M line not the procedure line" |
| **Diagnosis code count** | "Maximum 4 diagnosis codes per claim" |
| **Attachment requirements** | "PWK segment required for claims requiring medical records" |

### How to Use a Companion Guide

1. Download from payer's provider portal.
2. Read the payer-specific instructions for each segment.
3. Update your claim scrubber with the payer's special requirements.
4. Route claims differently based on payer rules.
5. Test with the payer's testing environment before production.

## 10. Medicare 837 Requirements

### Medicare 837P (Professional)

- Standard: ASC X12N 837 v5010A1 (005010X222A1)
- Required segments include submitting provider taxonomy, referring provider for specific services
- Medicare assigns each provider specific billing configurations

### Medicare 837I (Institutional)

- Standard: ASC X12N 837 v5010A2 (005010X223A2)
- Additional segments for Value Codes, Occurrence Codes, Condition Codes
- Required bill type (CMS-1450 / UB-04 derived)

### Medicare Acknowledgment Flow

```
Provider -> MAC: 837 Submission
MAC -> Provider: TA1 (Interchange Ack) -- seconds to minutes
MAC -> Provider: 997 (Functional Ack) -- seconds to minutes
MAC -> Provider: 277CA (Claim Status) -- 24-48 hours
MAC -> Provider: 835 (Payment/Advice) -- 7-21 days
```

## 11. Common Submission Errors

### Top 837 Submission Errors

| Error | Description | Prevention |
|-------|-------------|------------|
| **Invalid/Missing Taxonomy** | Taxonomy code missing or invalid for provider specialty | Add taxonomy to provider setup, validate against NUCC |
| **NPI/TIN Mismatch** | NPI does not match TIN on payer enrollment | Verify NPI/TIN pair in payer's provider directory |
| **Missing Referring Provider** | NPI of referring/ordering provider not included | Add referring NPI in Loop 2317B |
| **Invalid POS** | POS code not valid for procedure or payer | Validate POS against procedure type and payer rules |
| **Invalid Modifiers** | Modifier not valid for code or missing required modifier | Apply scrubber modifier validation |
| **Invalid Diagnosis Codes** | ICD-10 code invalid, truncated, or not used in US | Validate ICD-10 against current code set |
| **Missing or Invalid Patient ID** | Patient ID (member number) does not match payer records | Verify eligibility before submission |
| **Invalid CLM Values** | Total charge or patient responsibility fields incorrect | Verify charge amounts and COB fields |
| **Duplicate Claim** | Claim submitted more than once for same date/service | Implement duplicate check at provider level |
| **Date Range Errors** | Service date outside coverage period, or invalid date sequence | Validate dates against patient coverage, ensure from-date <= to-date |

### Error Prevention Strategies

1. **Implement a claim scrubber**: Apply scrubber edits before submission.
2. **Validate provider enrollment**: Verify NPI/TIN enrollment with each payer quarterly.
3. **Update codes quarterly**: Download and apply quarterly code updates.
4. **Train billers**: Regular training on new codes and payer rules.
5. **Use companion guides**: Configure billing system with each payer's requirements.
6. **Monitor rejection reports**: Track and analyze rejection patterns.
7. **Test with payers**: Use test files before going live with new payers or new codes.

## 12. Agent Training: Q&A Pairs for EDI 837

**Q: What is the difference between the 837P and 837I transaction?**

**A:** The 837P (Professional) is used by physicians, suppliers, and professional providers for claims derived from the CMS-1500 form. It uses the SV1 segment for service lines and requires Loop 2400 for each procedure. The 837I (Institutional) is used by hospitals, skilled nursing facilities, and other institutional providers for claims derived from the UB-04 form. It uses the SV2 segment with revenue codes, requires bill type and patient discharge status, and can span multiple dates of service in a single claim line. The 837I version is 005010X223A2, while the 837P is 005010X222A1.

---

**Q: What does a 997 Functional Acknowledgment tell the submitter?**

**A:** The 997 Functional Acknowledgment tells the submitter whether the EDI file was technically valid from a syntax and structure perspective. It confirms that the ISA/IEA envelope, GS/GE functional group, and ST/SE transaction set were formatted correctly according to X12 standards. It reports syntax errors such as missing segments, invalid data types, invalid qualifiers, or control number errors. However, the 997 does NOT indicate that the claim will be paid -- it is strictly a technical acknowledgment. A claim can pass the 997 and still be denied for medical necessity or eligibility.

---

**Q: What is the purpose of the ISA segment in an 837 transaction?**

**A:** The ISA (Interchange Control Header) segment is the first segment of an EDI interchange. It serves as the envelope that contains the entire transmission. The ISA identifies the sender and receiver of the transmission, includes the date and time of transmission, specifies the EDI version and standards, includes a unique interchange control number that helps track transmissions, specifies whether the transmission is production or testing, and provides the interchange-level acknowledgments. The ISA must be paired with an IEA (Interchange Control Trailer) that has a matching control number.

---

**Q: What is the function of the BHT segment in the 837?**

**A:** The BHT (Beginning of Hierarchical Transaction) segment identifies the purpose and structure of the 837 transaction. It specifies the transaction purpose code (00 = Original, 18 = Replace, 19 = Cancel), the reference identification number for the transaction, the transaction date and time, and the hierarchical structure code. The BHT is the second segment in the transaction set (after ST) and establishes the overall context for the claim submission.

---

**Q: What information is included in the CLM segment?**

**A:** The CLM (Claim Information) segment contains the claim-level data: the patient control number (internal claim ID), the total claimed amount, the place of service code, the claim filing indicator code (type of insurance), whether the provider signature is on file, the assignment of benefits indicator, and whether the benefits are being assigned to the provider. The CLM segment is in Loop 2300 and is the primary claim header segment that links all service lines to a single claim.

---

**Q: How does the 837 handle diagnosis codes?**

**A:** The 837 uses the HI (Health Care Diagnosis) segment in Loop 2300 to report diagnosis codes. The HI segment uses qualifiers to identify different diagnosis types. The principal diagnosis is identified with qualifier ABF, the admitting diagnosis with ABJ, the patient's reason for visit with ABK, and other diagnoses with ABF (subsequent occurrences). Diagnosis codes are reported with the decimal point. For example, a principal diagnosis of M17.0 is reported as ABF:M170. Multiple HI segments may be present to report additional diagnoses. The 837P supports up to 12 diagnosis codes; the 837I supports more with occurrence codes.

---

**Q: What is a companion guide and why is it important?**

**A:** A companion guide is a document published by a payer that explains how the X12 837 standard should be implemented for that specific payer. It specifies which optional segments are required, which situational segments apply, which code values are accepted, and any payer-specific formatting rules. Companion guides are important because each payer may interpret the X12 standard differently. Without following the companion guide, a provider's claims may be rejected even though they are technically valid by the X12 standard. Providers should download and configure their billing systems with each payer's companion guide.

---

**Q: What happens when a claim is transmitted with a testing indicator in production?**

**A:** When a claim is transmitted with the testing indicator (ISA15 = T for Testing) but sent to the production receiver, the payer's system will typically reject the claim because the receiver should be the testing environment. Conversely, sending a production claim (ISA15 = P for Production) to the testing environment will also result in rejection. The ISA15 value must match the environment expected by the receiver. This is a common setup error when configuring a new connection with a clearinghouse or direct payer.

---

**Q: What is the purpose of the NM1*85 segment?**

**A:** The NM1*85 segment in Loop 2010AA identifies the billing provider. It includes the billing provider's name (last name, first name, middle initial), the NPI (National Provider Identifier) in the REF segment that follows, and the tax identification number (TIN) in the REF*EI segment. The NM1*85 is a critical segment because the payer uses this information to validate the provider's enrollment and determine payment rates. An error in this segment (wrong name, wrong NPI configuration) can result in claim rejection.

---

**Q: What is the difference between a batch submission and a real-time submission?**

**A:** In batch submission, multiple claims are grouped into a single 837 file and transmitted as a batch. The provider receives acknowledgments asynchronously (997 immediately, 277CA within hours or days). Batch is suitable for high-volume submitters and institutional claims. In real-time submission, a single claim is transmitted individually and the provider receives a processing response in seconds. Real-time is suitable for low-volume submitters, point-of-service verification, and urgent claims. Batch uses FTP/SFTP/AS2 protocols while real-time typically uses HTTPS web services.

---

**Q: What is the role of the REF*G1 segment in the 837?**

**A:** The REF*G1 (Prior Authorization Number) segment is used to report the prior authorization number for services that require pre-authorization. It is located in Loop 2300 (claim level) or Loop 2400 (service line level). Claims for services that require prior authorization without this segment will be denied. A single claim may have one claim-level authorization number (in Loop 2300) and different line-level authorization numbers (in Loop 2400) if different services require different authorizations.

---

**Q: What does a 277CA response indicate about a claim?**

**A:** A 277CA (Claim Status) response provides the claim-level processing status from the payer. It indicates whether the claim was accepted into the payer's processing system, rejected due to format errors, is pending (in queue), has been paid, denied, or partially paid. The 277CA is a more meaningful response than the 997 because it provides claim-level status rather than file-level syntax validation. Providers typically receive a 277CA within 24-48 hours of submission, with a final status after adjudication (5-30 days).

---

**Q: What is required in the SV1 segment?**

**A:** The SV1 (Professional Service Line) segment is the primary service line detail in the 837P. It requires: the composite medical procedure identifier (CPT or HCPCS code with qualifier HC), the line item charge amount, the unit or basis for measurement (typically UN for units), the quantity of units billed, and optionally, up to four modifiers can be appended to the procedure code. The SV1 also supports a secondary procedure identification for reporting if the procedure code changes.

---

**Q: How is billing provider taxonomy submitted in the 837P?**

**A:** Billing provider taxonomy is submitted in the PRV (Provider Information) segment in Loop 2000A. The PRV segment uses qualifier BI (Billing Provider), followed by the taxonomy code (e.g., 207R00000X for Internal Medicine), and the taxonomy code qualifier code. The taxonomy code identifies the provider's specialty and determines the fee schedule applied. Some payers require the taxonomy code in the PRV segment, while others require it in the NM1 segment or in both places. The companion guide specifies the required location.

---

**Q: What does the HL segment represent in the 837 transaction?**

**A:** The HL (Hierarchical Level) segment establishes the hierarchical relationships between entities in the claim. The 837 has three hierarchical levels: HL1 (Billing Provider), HL2 (Subscriber), and HL3 (Dependent, if the patient is not the subscriber). Each HL segment includes an HL ID number, the parent HL ID number (to establish the hierarchy), the hierarchical child code (indicates whether there are child levels below), and the level code (20 = Information Source, 21 = Information Receiver, 22 = Subscriber, 23 = Dependent). The HL structure tells the payer who is being billed, who the subscriber is, and who the patient is.

---

**Q: What happens if the ISA13 control number is a duplicate of a previously submitted transmission?**

**A:** If the ISA13 (Interchange Control Number) is a duplicate of a previously submitted transmission within the same timeframe (typically 365 days), the receiving system will reject the transmission as a duplicate. This prevents accidental resubmission of the same file. The provider must use unique ISA13 numbers for each transmission. Many practice management systems auto-increment the control number for each file.

---

**Q: What is the proper way to report a service line with a modifier in the 837P?**

**A:** In the 837P, modifiers are reported in the SV1 segment as part of the composite medical procedure identifier. The procedure code is followed by modifiers separated by colons. For example: `SV1*HC:99213:25*` reports CPT 99213 with modifier 25. Up to four modifiers can be reported in the SV1 segment. If more than four modifiers are needed, modifier 99 (Multiple Modifiers) is used in the SV1 segment, and the additional modifiers are reported in the HCP segment at the service line level.

---

**Q: How does a provider cancel a previously submitted 837 claim?**

**A:** A provider can cancel a previously submitted 837 claim by submitting a new 837 transaction with the BHT segment purpose code set to 19 (Cancel). The cancellation 837 must include the original claim number in the REF*D9 segment (Payer Claim Control Number) and the CLM segment with the original claim information. The provider should not resubmit a new claim for the same service while the cancellation is processing. Some payers also support cancellation via the 820 transaction or a direct payer portal.

---

**Q: What is the role of the DTP segment in the 837?**

**A:** The DTP (Date/Time Period) segment reports dates associated with the claim or service line. Different DTP qualifiers identify different date types. Common DTP qualifiers include: 434 (Admission Date), 435 (Discharge Date), 472 (Service Date From), 473 (Service Date To), 096 (Onset of Symptoms), 431 (Last Seen Date), and 454 (Date of Service). The DTP segments are located in Loop 2300 for claim-level dates and Loop 2400 for service line dates. Dates are typically formatted as CCYYMMDD.

---

**Q: What is a TA1 segment in the 837 acknowledgment flow?**

**A:** A TA1 (Interchange Acknowledgement) is an acknowledgment segment that confirms receipt of the EDI interchange at the envelope level. It indicates whether the ISA segment was accepted or rejected (interchange-level validation). The TA1 is important because it tells the submitter whether the entire interchange was accepted or rejected at the outermost envelope level. If the ISA is rejected, none of the functional groups or transaction sets within that interchange are processed. TA1 acknowledgments are typically returned within minutes.

---

**Q: How does the 837 handle Coordination of Benefits?**

**A:** The 837 handles COB through the SBR (Subscriber Information) segment in Loop 2000B. The SBR segment includes the payer responsibility sequence code (P = Primary, S = Secondary, T = Tertiary, U = Unknown), the individual relationship code (18 = Self, 19 = Spouse, 20 = Child), and other subscriber-related information. For secondary claims, the claim must include the primary payer's payment information in the AMT segment and the primary payer's allowed/paid amounts. The DTP*573 segment reports the date of the primary payment.

---

**Q: What is a duplicate claim rejection?**

**A:** A duplicate claim rejection occurs when a payer receives a claim that matches an already-submitted claim based on a predefined set of criteria (typically provider, patient, date of service, and procedure code). The payer's system identifies the new claim as a duplicate and rejects it or denies it. Duplicate claims can be caused by accidental resubmission, clearinghouse reprocessing, multiple submission paths (e.g., clearinghouse and direct payer portal), or system errors. Providers should check claim status before resubmitting.

---

**Q: What is the proper way to replace a previously submitted 837 claim?**

**A:** To replace a previously submitted 837 claim, the provider submits a new 837 transaction with the BHT segment purpose code set to 18 (Replace). The replacement claim must include the original claim number in the REF*D9 segment and the updated claim information. The payer processes the replacement claim and voids the original claim. Providers should ensure that the replacement claim is complete -- the payer does not merge the replacement with the original; the replacement stands on its own.

---

**Q: What is the maximum number of diagnosis codes that can be submitted on a 837P claim?**

**A:** The 837P supports up to 12 diagnosis codes on a single claim. The principal diagnosis (qualifier ABF) is required. Up to 11 additional diagnosis codes can be reported with qualifier ABF in subsequent HI segments. The number of diagnosis codes the payer actually processes may be lower than 12. Some payers only process the first 4 diagnosis codes. Providers should list diagnoses in order of clinical relevance, with the primary reason for the encounter first and supporting/comorbid conditions thereafter.

---

**Q: How does a provider know if their 837 file was successfully received by the payer?**

**A:** The provider knows their 837 file was successfully received when they receive a TA1 or 997 Functional Acknowledgment. The TA1 confirms the interchange envelope (ISA/IEA) was received and validated. The 997 confirms the functional groups and transaction sets were received and syntactically valid. If neither acknowledgment is received within a reasonable timeframe (typically 24-48 hours), the provider should check with the clearinghouse or payer to confirm receipt. The absence of an acknowledgment does not necessarily mean the file was not received.

---

**Q: What is the purpose of the NM1*QC segment in the 837?**

**A:** The NM1*QC segment in Loop 2010BB identifies the patient (the person who received the service) when the patient is not the same as the subscriber (the insured party). For example, when a child is covered under a parent's insurance, the NM1*IL identifies the parent (subscriber/insured), and the NM1*QC Loop 2020A identifies the child (dependent/patient) with qualifier QC. If the patient is the subscriber, the NM1*QC loop is not required because the subscriber is the patient.

---

**Q: What is a "claim filing indicator" code in the CLM segment?**

**A:** The claim filing indicator code in the CLM05-1 element identifies the type of insurance or the entity the claim is being submitted to. Common values include: 12 = Medicare Part A, 13 = Medicare Part B, 14 = Medicaid, 15 = CHIP, 16 = TRICARE, 17 = CHAMPUS, 18 = Group Health Plan, 19 = FECA/Black Lung, 20 = Blue Cross Blue Shield, 21 = HMO, 22 = PPO, 23 = POS Plan, 24 = Indemnity Insurance, 25 = Auto Insurance, and many others. The code tells the payer which benefit plan rules to apply.

---

**Q: How does the 837 handle institutional bill types?**

**A:** In the 837I, the bill type is reported in the CLM05-1 element (the first component of CLM05, which also includes CLM05-2 for the frequency code). The bill type is a 3-digit code. For example, 111 = Hospital Inpatient (11) + Part A (1) + Admit Through Discharge Claim (1). The frequency code in CLM05-2 further specifies the claim type (0 = Non-payment/Zero Claim, 1 = Admit Through Discharge, 2 = Interim First Claim, 3 = Interim Continuing Claim, 4 = Interim Last Claim, 5 = Late Charges Only, 7 = Replacement of Prior Claim, 8 = Void of Prior Claim).

---

**Q: What is the purpose of the NM1*P4 segment in the 837?**

**A:** The NM1*P4 segment in Loop 2010CA identifies the provider who referred the patient (the referring provider). This segment is required for certain services such as diagnostic imaging, laboratory services, physical therapy, and specialty consultations. The referring provider's NPI is reported in the NM1 segment and the specialty/taxonomy in the PRV segment. Many payers require the referring provider NPI for all services that require a referral. Missing or invalid referring provider information is a common cause of claim rejection.

---

**Q: What is the proper format for reporting ICD-10 diagnosis codes in the 837?**

**A:** ICD-10 diagnosis codes are reported in the 837 with the decimal point omitted. For example, M17.0 is reported as M170, S06.5X9A is reported as S065X9A, and E11.51 is reported as E1151. The code is preceded by a qualifier such as ABF (principal diagnosis), ABJ (admitting diagnosis), or ABK (patient reason for visit). Diagnosis codes are case-sensitive in some systems -- typically uppercase. The code must be valid for the date of service and must be in the current ICD-10 code set.

---

**Q: How does a provider submit an attachment with an 837 claim?**

**A:** Attachments are submitted through the PWK (Paperwork) segment in the 837. The PWK segment identifies the type of attachment (such as medical records, X-rays, or lab results), the transmission method (BM = by mail, EL = electronically), and the attachment control number. If the attachment is submitted electronically, it may be submitted through a separate electronic attachment transaction or through a clearinghouse's document upload feature. The PWK segment alerts the payer that supporting documentation exists. Some payers require attachments before processing the claim.

---

**Q: What is the maximum number of service lines in a single 837 transaction?**

**A:** There is no hard X12 limit on the number of service lines in a single 837 transaction. However, practical limits exist based on the clearinghouse and payer specifications. Some clearinghouses limit file sizes to a certain number of claims or megabytes. Most practice management systems allow thousands of service lines per transmission. The provider should check with their clearinghouse and each payer for recommended file size limits. Very large files may cause timeout or processing issues.
